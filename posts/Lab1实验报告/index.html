1.思考题

Thinking 1.1

在阅读 附录中的编译链接详解 以及本章内容后，尝试分别使用实验环境中的原生 x86 工具链（gcc、ld、readelf、objdump 等）和 MIPS 交叉编译工具链（带有mips-linux-gnu- 前缀，如 mips-linux-gnu-gcc、mips-linux-gnu-ld），重复其中的编译和解析过程，观察相应的结果，并解释其中向 objdump 传入的参数的含义。

有关readelf.c文件有一些易错点首先是sh_table的获取，在ELF32_Ehdr结构体中e_shoff是以字节为单位的所以在加上它的时候要先把ehdr指针转化成char *类型sh_table = ehdr->e_shoff + (char *)ehdr;。（另外void *类型也是以一个字节为单位）

对于每一个文件都很好奇她到底在哪个位置，这有意义吗？可能是没有意义的，但是我还是喜欢搞清楚。

在 C 语言中，使用 extern 关键字声明的变量是外部变量，表示该变量在其他文件中定义（注意区分定义与声明）。

作为一个库函数，puts 被大量的程序所使用。因此，每次都将其从源代码编译一遍实在太浪费时间了。puts 的实现其实早就被编译成了二进制形式，这就是为什么要有链接过程的存在的原因之一。

我还是不清楚链接过程到底在干什么

  • 原生x86工具链

首先是预处理gcc -E hello.c -o hello.i，C语言的Preprocessor将头文件的内容添加到了源文件中文件依旧可以阅读。

然后是只编译不链接gcc -g -c hello.c -o hello.o，其中的-g便于分析汇编代码时与源代码相对应。对二进制代码反汇编得到汇编代码objdump --section=.text --disassemble=main --source hello.o，--section选项指定反汇编的节为.text，--disassemble指定反汇编的函数为main，--source表示显示汇编代码与源代码的对应关系。

平时我们编译链接文件，是不同文件之间的链接

但是目前是.c文件和lib库文件的链接

反汇编结果如下此时puts函数没有具体的实现，即无法正确的跳转

最后编译链接gcc -g hello.c -o hello，反汇编objdump --section=.text --disassemble=main -source hello，可以看见main函数的地址发生了变化，puts函数的跳转地址也正确的替换了。

  • MIPS 交叉编译工具链

首先是预处理mips-linus-gnu-gcc -E hello.c -o hello.i，然后是只编译不链接mips-linux-gnu-gcc -g -c hello.c -o hello.o，反汇编 mips-linux-gnu-objdump --section=.text --disassemble=main --source hello.o这个反汇编结果比x86要好懂多了，不过还是看不懂。

最后是编译链接mips-linux-gnu-gcc -g -static hello.c -o hello然后反汇编 mips-linux-gnu-objdump --section=.text --disassemble=main --source hello结果如下，肉眼可见地址发生了变化。

Thinking 1.2

思考下述问题：

• 尝试使用我们编写的 readelf 程序，解析之前在 target 目录下生成的内核 ELF 文件。

• 也许你会发现我们编写的 readelf 程序是不能解析 readelf 文件本身的，而我们刚才介绍的系统工具 readelf 则可以解析，这是为什么呢？（提示：尝试使用 readelf -h，并阅读 tools/readelf 目录下的 Makefile，观察 readelf 与 hello 的不同）

不管是编译得到的目标文件（relocatable）还是链接得到的了执行文件（executable）都具有相同的ELF结构。

ELF文件中存储了一堆结构体，其中有关段头（Phdr）和节头（Shdr）的信息都存储在ELF文件头结构体中（Ehdr）。

首先是用我们自己编写的readelf可执行文件解析mos得到每个节头的地址信息如下，一共有18个节

尝试用readelf解析自己输入./readelf readelf之后没有输出任何的结果，输入readelf -S readelf正常输出了readelf可执行文件的每个节头的信息。那为什么readelf无法解析身为ELF文件的自己呢？

查看tools/readelf目录下的Makefile可以看到再编译链接生成hello的时候命令加了3个额外的选项：-m32生成32位程序，即使机器是64位，也会按32位目标来编译和链接；-static使用静态链接，程序依赖的库会尽量直接打进可执行文件里，而不是运行时再去找动态库.so；-g在生成的可执行文件中加入调试信息，方便gdb调试。

少了一些选项得到的readelf可执行文件肯定和hello不同，但为什么无法被readelf解析我们就要从生成readelf可执行文件的源文件main.c和readelf.c入手，先来看main.c这个文件是负责把二进制文件加载到内存同时打印错误信息，由于./readelf readelf没有输出任何错误信息说明问题不出在main.c函数。

再来看我们的readelf.c文件，它定义了readelf函数打印节头地址返回整数。函数体内有这样以段代码Elf32_Ehdr *ehdr = (Elf32_Ehdr *)binary;它将传入的文件地址转化成ELF32_Ehdr类型，但是编译链接的时候如果不加入-m32得到的是64位的ELF文件（也可以直接通过readelf -h查看ELF头得知32/64位信息），函数在强制转化后解析错误，得到错误的shnum无法进入循环直接退出。

（多说一嘴，这里binary本应该是char *类型的指针，为什么可以转化成ELF32_Ehdr *类型？还有为什么会解析错误，指针的解析到底是什么原理？）

内核文件到底是什么？是一段二进制代码，这段代码表示的是一堆结构体，是按照elf.h文件中定义的结构体组织的文件。

Thinking 1.3

在理论课上我们了解到，MIPS 体系结构上电时，启动入口地址为 0xBFC00000（其实启动入口地址是根据具体型号而定的，由硬件逻辑确定，也有可能不是这个地址，但一定是一个确定的地址），但实验操作系统的内核入口并没有放在上电启动地址，而是按照内存布局图放置。思考为什么这样放置内核还能保证内核入口被正确跳转到？（提示：思考实验中启动过程的两阶段分别由谁执行。)

MIPS 上电后的固定启动地址对应的是第一阶段启动代码的入口，并不要求操作系统内核本身放在这里。在 MOS 实验中，第一阶段启动由 QEMU 提供的 bootloader 功能完成。QEMU 会直接加载 ELF 格式的内核，按照链接脚本和内存布局把内核放到指定地址，然后跳转到内核入口 _start。之后才进入第二阶段，即内核自己的启动代码：start.S 中的 _start 先完成 CPU 和栈初始化，再跳转到 mips_init。因此，虽然实验内核没有放在上电启动地址，而是按内存布局图放置，但仍然能够被正确跳转到并开始执行。

$(modules) 包含了 lib，init 和 kern 三个构建目标，分别对应操作系统的一些依赖库，初始化代码和内核代码

有关printk函数

独立环境和宿主环境，独立环境不提供I/O操作。

要弄懂内核如何将信息输出到控制台，需要阅读以下三个文件：kern/printk.c，lib/print.c和 kern/machine.c。

先来看看kern/printk.c函数，其中有一个printk函数，它有变长参数列表，获取参数必须要使用stdarg.h头文件定义的变量和宏。

printf(“The answer is %d,%s”,n,str);

参数fmt除了包含直接输出的字符，还可以包含格式符（format specifiers），原型如下：%[flags][width][length]<specifier>

2.难点分析

这次上机算是顺利的过了extra和exam，看来还是要失败一次才能找到经验。单纯从技巧的角度来说，上机其实比较考验阅读能力和理解能力，能够在一堆代码中找到自己真正要干的事情。题目中有很多比如说“题目背景”、“前置知识”等可以大略地看一下，因为很多都是你知道的东西，要重点关注的是”题目要求“、“评测标准”和“提示”。（有时候助教们的好心反而加大了题目难度）

lab1-exam

打印每个节的节名称字符串，涉及到了节名称字符串表，虽然它是一个表但是它的存储形式和一般的节没什么区别，都是要有一个节头来指向。

节头表（Section Header Table）存储了很多节头（Section Header），其中Elf64_Shdr shstrShdr就是其中的一个节头，它在节头表里面的偏移由ELF头里面Elf64_Half e_shstrndx记录。在拿到指向存储了所有节名称字符串的节头指针shstrShdr后我们就要在ELF文件中找到内容所在地，也就是利用sh_offset计算出所有字符串的起始地址shstrtab*。（所有字符串的内容大致为\0.text\0.data\0.bss\0.symtab...\0）

最后取出节名称字符串的操作利用到了数组和结构体 sectionName = shstrtab + sh_table[i].sh_name;。

sectionHeader1 sectionHeader2 sectionHeader3 …… sectionHeader(k-1) sectionHeader(k)

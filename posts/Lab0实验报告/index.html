上机实验

上机实验我犯了很多非常傻的错误，包括最为致命的把题目的127.0.0.3这个IP的替换当作了要把所有的log/$date/access.log替换掉。

我觉得我这次上机extra没有做出来主要是因为熟练度不够，同时不知道到底要考什么，所以在遇到一些问题之后就慌了神，即使脑袋里有印象该怎么做，最后还是做不出来。往后的上机要做好预案，在上机之前就想好上机的时候可能遇到什么问题，有哪些可能的解决手段（指的是不通过互联网的解决手段，比如携带纸质资料，上传资料到跳板机）

有关wordspliting

题目有要求把grep命令找到的含有ERROR的行全部原封不动的输入到reports/date文件夹下的error.log中，这时我们很容易想到echo命令，但是我们不能直接echo $errors > $error_dst因为echo会在输出的时候做分词，会把所有的空白符转换成空格，为了抑制分词可以在$errors外面套一层""变成echo "$errors" > $error_dst。

MakeFile有关depends

我在上机的时候遇到了神奇的报错make: *** 没有规则可制作目标“common.h,”，由“ver1” 需求。 停止。

后来找了半天才发现是在写depends的时候手贱加上了,导致无法正确的解析出来target的依赖。

还有就是有关rm命令，一般来说如果你要删除一个不存在的文件的时候rm命令会报错，这个时候就需要加上-f

选项，这样在没有文件的时候删除也不会报错了。（我要狠狠的吐槽助教为什么稿那么多的宏变量在文件里面，我光是要理解就花了好长的时间）

bash有关的操作

1.mkdir如何避免重复创建目录而报错

在创建目录的时候经常会遇到重复创建而报错的时候，这就需要添加参数-p来避免报错了，也就是mkdir -p mewdir

2.sort的使用方法

sort默认是按行切分，然后对每行逐个字符的比较，最后按照字符串从小到大来输出。-n选项是按照数字排序，具体是把每一行开头能够识别的数字识别出来然后比较。-r选项表示reverse，指的是反向排序。

3.正则表达式Regex

虽然题目中没有很复杂的正则表达式但是由于笨人理解能力太过于低下，导致我陷入了如何匹配类似于127.0.0.3这样的IP地址的问题中，当然我是不可能在考场上做出来的，所以课下研究了一下。这最好是一个非贪婪匹配，但是由于整个log符合条件的串少之又少，所以我就不管贪不贪婪了。首先这个的格式是num + . + num + . + num + . + num但是num的个数不确定，所以我们就要用到* -> 0次或者多次或者+ -> 1次或者多次来指定重复次数。同时由于有些字符比如.在sed指令中本身就有含义，所以为了正确表达，必须要加上反斜杠\来转义，最后的表达式如下sed 's/\([0-9]\+\.\)\{3\}[0-9]\+' file。

线上实验算是比较顺利，没有遇到什么比较难的地方

Thinking 0.1

思考下列有关 Git 的问题：

• 在前述已初始化的 ~/learnGit 目录下，创建一个名为 README.txt 的文件。执行命令 git status > Untracked.txt（其中的 > 为输出重定向，我们将在 0.6.3 中详细介绍）。

• 在 README.txt 文件中添加任意文件内容，然后使用 add 命令，再执行命令 git status > Stage.txt。

• 提交 README.txt，并在提交说明里写入自己的学号。

• 执行命令 cat Untracked.txt 和 cat Stage.txt，对比两次运行的结果，体会README.txt 两次所处位置的不同。

• 修改 README.txt 文件，再执行命令 git status > Modified.txt。

• 执行命令 cat Modified.txt，观察其结果和第一次执行 add 命令之前的 status 是否一样，并思考原因。

执行git status > Untracked.txt ，表示查询当前README.txt文件状态，并将其记录在Untracked.txt文件中；git status > Stage.txt和git status > Modified.txt 同理。
cat Untracked.txt后，第二行显示Untracked files:，说明：在README.txt新建的时候，其处于为未跟踪状态 （untracked）；
cat Stage.txt后，第二行显示 Changes to be committed:，说明：在README.txt中任意添加内容，接着用 add 命令之后，文件处于暂存状 态（staged）；
cat Modified.txt后，第二行显示 Changes not staged for commit:，说明：在修改README.txt之后，其处于被修改状态（modified）。

add the file : git add

stage the file : git add

commit : git commit -m ''

Thinking 0.3

思考下列问题：

  1. 代码文件 print.c 被错误删除时，应当使用什么命令将其恢复？
  2. 代码文件 print.c 被错误删除后，执行了 git rm print.c 命令，此时应当
    使用什么命令将其恢复？
  3. 无关文件 hello.txt 已经被添加到暂存区时，如何在不删除此文件的前提下
    将其移出暂存区？

1.git checkout -- print.c

2.git rm不仅仅会删除暂存区，还会删除磁盘，所以要使用git reset HEAD print.c && git checkout -- print.c

3.git rm --cached hello.txt
